
## Контекст

В нашей системе при нажатии кнопки `Принять` подтвержденный диагноз должен быть передан в ИСБП.

Интеграция нужна для создания записи инцидента в ИСБП и последующего контроля ее финального статуса.

Особенность бизнес-процесса ИСБП:

- первичный `POST` не завершает жизненный цикл инцидента;
- `POST` только создает запись инцидента и частично автозаполняет ее;
- затем врач-эксперт в ИСБП открывает запись, дозаполняет опросник и сохраняет ее;
- только после этого запись считается окончательно подтвержденной и попадает в регистр ИСБП.

Из-за этого интеграция должна поддерживать минимум два этапа:

1. запись создана в ИСБП;
2. запись подтверждена в ИСБП после действий врача-эксперта.

---

## Бизнес-логика

### Момент запуска интеграции

Интеграция запускается в момент подтверждения диагноза действием `Принять`.

Базовая сущность интеграции на нашей стороне:

- `DiagnosisState`

### Ключевые идентификаторы

В интеграции должны использоваться два разных идентификатора:

| Идентификатор | Роль |
|---------------|------|
| `mh_rn` | основной идентификатор случая для ИСБП |
| `diagnosis_state_id` | идентификатор нашей системы для трекинга, `GET`-запроса и будущей связки сущностей |

### Общий сценарий

1. Пользователь подтверждает диагноз.
2. Наша система формирует payload.
3. В ИСБП выполняется `POST` запрос на создание записи инцидента.
4. Если ИСБП приняла запрос и создала запись, в нашей системе фиксируется промежуточный интеграционный статус.
5. Далее наша система или фоновый процесс делает `GET` запрос статуса.
6. После того как врач-эксперт в ИСБП дозаполнил запись и сохранил ее в регистре, `GET` должен вернуть финальный статус.
7. После этого в нашей системе фиксируется финальный интеграционный статус.

## POST запрос

### Назначение

Создание записи инцидента в ИСБП на основании подтвержденного диагноза.

### Endpoint

```text
POST https://test-aipss.cito-priorov.ru/vbo/create_from_incidents
```

### Авторизация

В запросе обязательно должен передаваться статичный Bearer token:

```http
Authorization: Bearer <static_token>
```


### Назначение body

В теле запроса должны передаваться:

- идентификаторы случая и диагноза;
- данные пациента;
- данные диагноза;
- данные врача-эксперта;
- список событий, участвующих в формировании диагноза;
- служебные поля трассировки.

### Обязательные поля POST

| Поле | Тип | Обязательность | Комментарий |
|------|-----|----------------|-------------|
| `mh_rn` | `integer` | да | главный идентификатор случая для ИСБП |
| `diagnosis_state_id` | `integer` | да | идентификатор диагноза в нашей системе |
| `comment` | `string` | да | комментарий/обоснование отправки диагноза/инцидента в ИСБП по кнопке `Принять` |
| `diagnosis.id` | `integer` | да | ID типа диагноза |
| `diagnosis.title` | `string` | да | наименование диагноза |
| `diagnosis.status` | `string` | да | ожидается `confirmed` |
| `patient.patient_name` | `string` | да | ФИО пациента |
| `user` | `object` | да | данные врача-эксперта |
| `user.id` | `integer|string` | да | внутренний идентификатор пользователя |
| `user.display_name` | `string` | да | ФИО врача-эксперта |
| `events` | `array` | да | список событий, участвовавших в формировании диагноза |

### Поля POST, рекомендуемые к передаче

| Поле | Тип | Комментарий |
|------|-----|-------------|
| `patient.amb_card_number` | `string` | номер амбулаторной карты |
| `patient.external_patient_id` | `integer` | внешний ID пациента, если есть |
| `patient.birth_date` | `date` | дата рождения |
| `patient.sex` | `integer` | пол |
| `stac_card.card_number` | `string` | номер стац-карты |
| `stac_card.mht_numb` | `string` | дополнительный номер |
| `stac_card.department` | `integer` | отделение |
| `stac_card.admission_date` | `datetime` | дата поступления |
| `stac_card.discharge_date` | `datetime` | дата выписки |
| `stac_card.weight` | `number` | вес |
| `stac_card.height` | `number` | рост |
| `user.ad_username` | `string` | логин пользователя |

### События

В `POST` должны передаваться события, которые участвовали в формировании конкретного подтвержденного диагноза.

Источник связи на нашей стороне:

- `DiagnosisState`
- `DiagnosisEventState`
- `Event`

В ИСБП передаются все основные поля событий, кроме `details`.

То есть:

- `details` не передаются;
- полная детализация событий в первом этапе интеграции не требуется;
- при необходимости это может быть расширено отдельным согласованием.

### Рекомендуемый ответ POST

```json
{
  "success": true,
  "status": "CREATED_DRAFT",
  "message": "Запись инцидента создана в ИСБП",
  "mh_rn": 456789,
  "diagnosis_state_id": 1254,
  "isbp_incident_id": "ISBP-2026-00001234"
}
```

### Семантика ответа POST

| Поле | Тип | Комментарий |
|------|-----|-------------|
| `success` | `boolean` | успешно ли обработан запрос |
| `status` | `string` | текущий статус записи в ИСБП |
| `message` | `string` | текстовое пояснение |
| `mh_rn` | `integer` | идентификатор случая |
| `diagnosis_state_id` | `integer` | идентификатор диагноза в нашей системе |
| `isbp_incident_id` | `string/null` | ID записи в ИСБП, если ИСБП его возвращает |

### Что считать успешным результатом POST

Успешный `POST` означает:

- ИСБП приняла данные;
- в ИСБП создана запись инцидента;
- запись еще может быть не подтверждена в регистре.

## GET запрос

### Назначение

Получение текущего статуса записи инцидента, ранее созданной в ИСБП.

### Статус endpoint


### Рекомендуемый endpoint

```text
GET https://test-aipss.cito-priorov.ru/vbo/incidents/status?mh_rn={mh_rn}&diagnosis_state_id={diagnosis_state_id}
```

или как альтернативный вариант:

```text
GET https://test-aipss.cito-priorov.ru/vbo/incidents/{diagnosis_state_id}/status?mh_rn={mh_rn}
```

### Требование к GET

`GET` должен позволять однозначно определить:

- найдена ли запись в ИСБП;
- создана ли только черновая запись;
- подтверждена ли запись после дозаполнения врачом;
- произошла ли ошибка;
- отсутствует ли запись по переданным идентификаторам.

### Рекомендуемый ответ GET

```json
{
  "success": true,
  "status": "CONFIRMED",
  "message": "Запись подтверждена и сохранена в регистре ИСБП",
  "mh_rn": 456789,
  "diagnosis_state_id": 1254,
  "isbp_incident_id": "ISBP-2026-00001234",
  "confirmed_at": "2026-04-20T12:35:10Z"
}
```

### Семантика ответа GET

| Поле | Тип | Комментарий |
|------|-----|-------------|
| `success` | `boolean` | успешно ли обработан запрос статуса |
| `status` | `string` | текущий статус записи в ИСБП |
| `message` | `string` | текстовое пояснение |
| `mh_rn` | `integer` | идентификатор случая |
| `diagnosis_state_id` | `integer` | идентификатор диагноза в нашей системе |
| `isbp_incident_id` | `string/null` | идентификатор записи в ИСБП |
| `confirmed_at` | `datetime/null` | дата финального подтверждения в ИСБП |

## Статусы

### Бизнес-статусы на стороне ИСБП


| Статус | Смысл |
|--------|-------|
| `CREATED_DRAFT` | запись создана в ИСБП, но не подтверждена в регистре |
| `IN_PROGRESS` | запись существует и находится в работе у врача |
| `CONFIRMED` | запись дозаполнена и подтверждена в ИСБП |
| `REJECTED` | запись отклонена или не может быть завершена |
| `ERROR` | ошибка обработки |
| `NOT_FOUND` | запись по идентификаторам не найдена |

### Интеграционные статусы на нашей стороне

С учетом двухэтапного процесса рекомендуется использовать отдельные интеграционные состояния, логически связанные с `DiagnosisStateStatus`.

Рекомендуемый вариант:

| Статус в нашей системе | Условие |
|------------------------|---------|
| `Отправлено в ИСБП` | успешный `POST`, запись создана в ИСБП |
| `Добавлено в ИСБП` | `GET` вернул финальный подтвержденный статус |

### Уточнить

| Вопрос | Статус |
|--------|--------|
| точный `GET` endpoint | нужно согласовать с ИСБП |
| финальный справочник статусов ИСБП | нужно согласовать с ИСБП |
| финальный справочник ошибок и `error_code` | нужно согласовать с ИСБП |

| точный состав полей пользователя, обязательных для ИСБП | нужно уточнить |
